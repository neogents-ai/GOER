Building the SFS-OS Seed Agent — Nori

Stack: Python, SQLite, Claude API (via OpenRouter), SFS-OS  |  Domain: AI Infrastructure / Python Agent Development

What This Is

SFS-OS is a self-hosted Python multi-agent system that runs weekly business and personal life audits. The seed agent (Nori) is the orchestrator — it boots the system, routes data, dispatches 9 specialist sub-agents, writes briefs, and persists findings to a SQLite database via db.py. This playbook covers how the full pipeline was built: the agent architecture, the database layer, and how everything is wired together.

Step 2 — Build the database layer (db.py)

Before the seed agent, build the persistence layer. Two tables: posture_items for security/IT findings that survive across runs, and meter_events for API usage tracking.

# lib/db.py — key functions

init_db()
# Creates both tables if they don't exist. Safe to call every run.
# Called at the top of seed.py main() so tables always exist before agents run.

upsert_posture_item(domain, agent, title, detail, severity, source_file, run_date)
# Inserts a finding. If an open item with the same domain+agent+title already
# exists, updates it instead of creating a duplicate. Prevents finding bloat.
# Severity values: critical | high | medium | low | info

resolve_posture_item(item_id, status)
# Marks a finding resolved. Status: closed | accepted | wont_fix

get_posture_summary()
# Returns counts by severity for all open items.
# Used by CISO agent dashboard to show: 3 critical, 1 high, etc.

log_meter_event(agent, model, prompt_tokens, completion_tokens, latency_ms, success)
# Logs every API call. Lets you track spend per agent per run.

Step 3 — Wire db.py into seed.py

The key connection: init_db() is called once at startup, before any agents run. This guarantees tables exist. The CISO agent then writes findings to posture_items via upsert_posture_item() during its run.

# In seed.py main():

from lib.db import init_db

def main():
    config = load_config()
    client = get_client()

    # Initialize database — creates tables if they don't exist
    init_db()

    # Now safe to run agents that write to the DB
    emit_event("plan", "active", config)
    ...
Why upsert instead of insert? The CISO agent runs every week. Without upsert, you'd get duplicate rows for the same open finding (e.g., "SSH key rotation overdue") accumulating every run. Upsert checks for an existing open item with the same domain+agent+title — if found, it updates the detail and timestamp instead of creating a new row.

Step 4 — Build the seed agent orchestration flow

Nori's decision filter runs before dispatching agents — it checks what data is available and prints a readiness report:

def run_decision_filter(business_manifest, personal_manifest):
    has_biz = "YES" if has_data(business_manifest, "csv", "text") else "NO — add business data files"
    print(f"  1. Revenue support in 30 days?  {has_biz}")
    print(f"  2. Existing team member handles?  NO — Nori is orchestrating")
    print(f"  3. Coach Berry bottleneck?  CHECKING — milestone agent will assess")
    print(f"  4. Broken manual process?  CHECKING — risk agent will assess")

Agents are gated on data availability — some always run, some skip if no data is present:

# Revenue only runs if there's actual data to analyze
if has_data(manifest, DATA_TYPE_CSV, DATA_TYPE_TEXT):
    results.append(revenue_agent.run(manifest, client, config))
else:
    skipped.append("revenue (no leads.csv or crm_notes.txt)")

# Milestone always runs — works from config even with no data files
results.append(milestone_agent.run(manifest, client, config))

Step 5 — Persistent memory between runs

After every run, Nori writes a memory file that gets loaded at the top of the next run. This gives the agent continuity — it knows last week's MRR, milestone progress, and open risks without re-reading all the data.

# Memory is written at end of main()
update_memory(all_results, config, persona_id)

# Memory is read at start of main()
memory = read_memory(persona_id)
if memory:
    print(f"[nori/{persona_id}] Memory loaded from last session.")

Memory file location: memory/nori-memory.md — human-readable markdown, versioned with each run date.

Step 6 — The inbox classifier (auto-routing)

Drop any file into data/inbox/ and the classifier figures out where it belongs. Bank statement? Routes to data/personal/. CRM export? Routes to data/business/. Uses Claude to classify by content, not filename.

# Called before scanning in main()
classify_inbox(BASE_DIR, client)

# After classification, scan picks up correctly routed files
biz_manifest = scan(str(_data_root(config) / "business"))
per_manifest  = scan(str(_data_root(config) / "personal"))
